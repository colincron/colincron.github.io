Hunting IDORs on GoDaddy Managed WordPress

During a routine NuScrape crawl, I found a UUID-based SSO endpoint sitting in public page source on a GoDaddy Managed WordPress site. Here's what IDOR means, why this finding is worth a closer look, and where I stopped.

Most interesting findings don't announce themselves. This one showed up as an XHR endpoint in NuScrape's database — a URL that looked slightly out of place. That's usually where the interesting stuff hides.

What Is an IDOR?

IDOR stands for Insecure Direct Object Reference. The concept is simple: an application gives you a URL that directly references an object — a user record, a file, an account — and then fails to verify whether you're actually authorized to access it.

The classic example is a URL like /user/profile?id=1042. If you can change 1042 to 1043 and see someone else's profile, that's an IDOR. The application used a predictable identifier and never checked whether the requesting user owned it.

IDORs consistently rank among the most common and highest-paid findings on bug bounty platforms because they're often simple to find, easy to demonstrate, and can have serious real-world impact — account takeovers, data exposure, privilege escalation.

The Finding

During a crawl of a GoDaddy Managed WordPress site, NuScrape flagged an XHR endpoint it hadn't seen before:

XHR endpoint — discovered in page source
https://host.godaddy.com/mwp/site/8d96ec13-297d-4796-87e1-8091c57a5107/sso
Source page: https://alliancechamber.com/wp-login.php
Method: GET (unauthenticated discovery)

A few things immediately stood out. First, the endpoint lives on host.godaddy.com — not the target site itself. This is GoDaddy's hosting infrastructure. Second, the path structure is /mwp/site/{uuid}/sso. The sso at the end means Single Sign-On — this endpoint is designed to automatically log someone into the site's WordPress admin without requiring a separate login step. Third, that UUID was sitting in the public HTML source of the login page, visible to anyone who viewed the page source or ran a crawler against it.

Why the UUID Matters

GoDaddy uses UUIDs as site identifiers across their Managed WordPress platform. Every site gets one. The UUID in this case — 8d96ec13-297d-4796-87e1-8091c57a5107 — is the unique identifier for alliancechamber.com's WordPress installation on GoDaddy's infrastructure.

UUIDs are long and look random, which gives a false sense of security. They're not passwords. They're not secret. This one was embedded in the public login page source, which means any automated scanner, any curious person, or any malicious actor could find it in about thirty seconds.

The question that matters for a bug bounty report is: what happens when you hit that SSO endpoint while authenticated as a different GoDaddy user?

The IDOR hypothesis If GoDaddy's SSO endpoint validates that the requesting user owns the site before granting access, this is a non-issue. If it only checks that the UUID exists and the requester has any valid GoDaddy session — regardless of which account — that's a cross-account authentication bypass. An attacker with any GoDaddy account could hit another site's SSO endpoint using that site's publicly discoverable UUID and land directly in the victim's WordPress admin.

Verifying the Exposure

I confirmed the endpoint was real and behaving as expected for an unauthenticated request:

curl -si "https://host.godaddy.com/mwp/site/8d96ec13-297d-4796-87e1-8091c57a5107/sso" \
  | head -10

The endpoint returned a redirect to GoDaddy's SSO flow — it's live and accepting requests. I also verified through NuScrape that different sites produce different UUIDs, meaning each is site-specific and the UUID is the only thing distinguishing one site's admin access endpoint from another's.

I also checked whether the UUID was unique to this site or a shared value across GoDaddy's platform:

sqlite3 ~/NuScrape/ScrapeDB \
  "SELECT url FROM XHREndpoints
   WHERE url LIKE '%mwp/site%'
   LIMIT 20;"

Different sites, different UUIDs. The identifier is site-specific and publicly exposed in page source on every GoDaddy Managed WordPress login page.

What I Confirmed vs. What I Didn't

confirmed

SSO endpoint exists and accepts unauthenticated requests. UUID is publicly discoverable in login page source. Different sites have different UUIDs. Endpoint redirects to GoDaddy SSO flow.

not yet verified

Whether an authenticated session from account A can access site B's SSO endpoint. This is the critical question — and testing it requires a second GoDaddy account and an authorized test site.

This is where responsible research has a ceiling. To fully verify the IDOR, I'd need to create a free GoDaddy account, spin up a trial Managed WordPress site to get my own UUID, then attempt to access the original site's SSO endpoint using my session. If it grants access to the wrong site's admin panel, that's a confirmed critical finding. If it rejects the request, the authorization check is working correctly.

I haven't done that test yet. Which means I have a strong hypothesis and solid supporting evidence, but not a confirmed exploit.

Important Testing this without authorization against a real site you don't own would be unauthorized access. The correct approach is to use your own test site as the target — never someone else's. Any testing should only be done against infrastructure you own or have explicit written permission to test.

Why It's Still Worth Reporting

Bug bounty programs — including GoDaddy's HackerOne program — accept reports where the researcher couldn't fully verify the impact themselves but has documented compelling evidence of a potential vulnerability. The security team can verify the authenticated half internally without any risk of unauthorized access.

What makes this finding worth reporting even unconfirmed:

The UUID exposure alone is a real issue. SSO endpoints that rely on a UUID as the sole access control mechanism are a design problem regardless of whether the cross-account bypass works. UUIDs are identifiers, not credentials, and treating them as a secret while embedding them in public page source is a contradiction.

If the bypass does work, the impact is severe. GoDaddy hosts millions of WordPress sites. A finding that lets any GoDaddy account holder access any other site's admin panel via a publicly discoverable identifier would be a critical severity finding — the kind that pays well on HackerOne and gets a CVE.

Next Steps

The plan is to create a free GoDaddy trial account and a test Managed WordPress site, then run the cross-account test against my own infrastructure. If it confirms the bypass, I'll write a full reproduction report for GoDaddy's HackerOne program with step-by-step instructions. If it doesn't, I'll still file a lower-severity report about the UUID exposure in public page source — because that design choice is worth flagging regardless.

I'll post a follow-up when I have results.

// key takeaways

  1. IDORs don't require brute force. This one was sitting in page source — NuScrape found it passively during a normal crawl with no aggressive scanning.
  2. UUIDs are not secrets. Long random-looking identifiers embedded in public HTML are discoverable by anyone. Don't use them as access controls.
  3. Partial findings are reportable. You don't need a full exploit chain to file a bug bounty report — documented evidence of a potential vulnerability is enough for the program's security team to investigate the rest.
  4. Know where to stop. Testing cross-account access requires your own test infrastructure. Never attempt to verify an IDOR against a real site you don't own.
