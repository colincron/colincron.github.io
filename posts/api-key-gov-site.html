How I Found an API Key Exposed on a Federal .gov Site

I wasn't looking for anything specific. I was running a routine crawl with a recon tool I've been building called NuScrape, pointed at a government job training site, and it flagged something I almost scrolled past.

A string. In a JavaScript file. That looked like an API key. Here's how it happened, what I did about it, and what it means.

The Setup

NuScrape is a web recon and vulnerability scanner I've been building for authorized red team research and bug bounty work. One of its core features is JavaScript analysis — after it crawls a site, it pulls down every JS file it finds and runs pattern matching against the content looking for things that shouldn't be public. API keys, tokens, credentials, internal endpoints.

The target was apprenticeship.gov, a U.S. Department of Labor site. I wasn't running an aggressive scan — just a standard crawl with JS analysis enabled.

The Finding

NuScrape flagged a value in config.js:

!! HIGH — JS finding: api_key
OEDzPHdEzLUCrDiMClKbwb-YasE3hez9bbZswBMKwu0
Source: https://www.apprenticeship.gov/react/static/js/config.js?1772722915
Found at: 10:53:56

43 characters, base64url encoded. NuScrape classified it as api_key. The file it came from — config.js — is significant. Config files are where developers intentionally put configuration values. Finding something that looks like a key in a config file is a different signal than finding it buried in minified application code.

The finding was stored in my SQLite database with the source JS URL, the page it was discovered from, and a timestamp.

Verifying Without Touching Anything

This is the part that matters most, especially on a .gov target.

Legal note The CFAA (Computer Fraud and Abuse Act) is not something to play with. Even a single authenticated API request using someone else's key — even "just to test if it works" — can cross a legal line. Passive discovery is fine. Active exploitation is not.

So the goal was to figure out what this key was without using it. First, I looked at the format. It didn't match AWS key patterns (those start with AKIA). It wasn't a JWT — those have two dots separating three base64 segments. The length and encoding were consistent with a Google API key or similar service token.

Next step was to fetch the config file directly and look at the surrounding context — what variable name it was assigned to, what service the code referenced nearby. That would identify the service without making any authenticated requests.

curl -s "https://www.apprenticeship.gov/react/static/js/config.js?1772722915" \
  | grep -o '.\{80\}OEDzPHdEzLUCrDiMClKbwb.\{80\}'

Empty output. The file had already been redeployed. By the time I went to verify, the key was gone from the live site.

What the Database Told Me

Even though the key was no longer in the live file, NuScrape had already stored the finding with a timestamp. Querying the database:

sqlite3 ~/NuScrape/ScrapeDB \
  "SELECT value, js_url, found_at FROM JSFindings
   WHERE value LIKE '%OEDzPHdEzLUCrDiMClKbwb%';"

The key appeared twice — captured from both config.js and main.js during the same crawl session at 10:53. The cache-busting timestamp in the URLs meant those were the exact file versions NuScrape saw at crawl time.

The fact that it appeared in both files suggested the config values were being bundled into the main application — a common pattern in React apps where environment variables get compiled into the build output and shipped to the browser.

What I Did With It

I documented what I had: the key value, the JS files it was found in with versioned URLs, the page it was discovered from, the timestamp of discovery, and the fact that the key was no longer present in live files when I went to verify.

For .gov sites, CISA has a vulnerability disclosure policy covering federal agencies. The reporting path is either the agency's own disclosure policy at /vulnerability-disclosure-policy or directly through cisa.gov.

The redeployment suggests someone was already working on the site when I found it — possible the key was already being rotated. But the right move is still to report it, note what you found and when, and let the agency's security team verify whether the key was ever active and whether rotation was handled cleanly.

Why This Matters

API keys in JavaScript bundles are one of the most common and most preventable exposures in web security. The reason it keeps happening is straightforward: during development, keys get hardcoded for convenience. The code gets committed, the key gets bundled into the frontend build, and it ships to production — where anyone with a browser and five minutes can find it.

For a government site, even a low-privilege key can be serious depending on what service it accesses. Read access to a database. Quota consumption. A key that authenticates to an internal API not designed to be called from outside.

The fix is simple: API keys don't belong in frontend code, ever. They belong in server-side environment variables, accessed through a backend the browser never directly touches.

// key takeaways

  1. Context matters more than the finding. A string that looks like an API key is not a confirmed vulnerability — you need to know what service, what permissions, what exposure before you can assess severity.
  2. The database is your evidence. Timestamped records from crawl time give you something to reference even after the live site changes.
  3. Passive recon has a ceiling. On a .gov target especially, passive discovery is where you stop. Document it, report it, move on.
  4. Automated JS analysis catches what manual review misses. The key was in a minified, cache-busted file buried in a build directory — there's no realistic way to catch this class of issue at scale without tooling.
