Building a Target Discovery Pipeline with Shodan and HackerOne's Public Program Data

Bug bounty hunting is a numbers game. I wanted a way to find exposed services on the internet and immediately know whether the target has a bug bounty program before spending any time on verification. This is how I built that pipeline — and what I learned about false positives along the way.

The Problem

Shodan is great for finding exposed services — Adminer instances, Kibana dashboards, Jenkins servers, open Redis ports. The problem is that most of what Shodan surfaces is hosted on generic VPS providers with no bug bounty program. You can spend hours verifying findings on targets that have no disclosure path.

What I needed was a way to cross-reference Shodan results against known bug bounty programs automatically, so I could filter down to targets worth investigating before touching anything.

The Data Source

HackerOne's public program list used to be accessible without authentication, but that endpoint is gone. What works now is bounty-targets-data, a community-maintained JSON dump of all public bug bounty programs including their in-scope domains. It updates daily and requires no API key.

curl -s "https://raw.githubusercontent.com/arkadiyt/bounty-targets-data/main/data/hackerone_data.json" | python3 -c "
import json,sys
data=json.load(sys.stdin)
print(f'Programs: {len(data)}')
print(f'First: {data[0][\"handle\"]} — {data[0][\"name\"]}')
"

The Script

The core idea is simple — pull Shodan results, extract hostnames, and check them against an index built from the bounty-targets-data. Here's the matching logic I landed on after some iteration:

def match(org, hostnames, domain_idx, name_idx):
    matches, seen = [], set()
    def add(m):
        if m["handle"] not in seen:
            seen.add(m["handle"]); matches.append(m)
    for h in (hostnames or []):
        for part in h.lower().split("."):
            if len(part) > 4:
                if part in domain_idx: add(domain_idx[part])
    return matches[:5]

The key lesson here was to match only on hostnames, not org names. Early versions matched on org name tokens — "technologies", "corporation", "group" — and produced garbage results. Microsoft Corporation matching Insulet Corporation because they share the word "corporation" is useless signal. Hostname fragment matching is noisier in theory but much cleaner in practice because hostnames are specific.

What It Looks Like in Practice

Running the tool against exposed Adminer instances:

python3 shodan_h1.py --query 'http.title:"Adminer" http.status:200' --limit 200 --h1-only

Out of 200 results, 43 had potential H1 program matches. Most were still noise — generic hosting tokens matching unrelated programs. But a handful had "adminer" explicitly in the hostname, which is the real signal:

✓ potential match
54.65.212.170:443 — Amazon Technologies Inc.
Hostnames: adminer-staging.shikigaku-cloud.com
H1: John Deere → https://hackerone.com/john-deere

The John Deere match there was a false positive — "deere" wasn't in the hostname. But adminer-staging.shikigaku-cloud.com is still worth noting, because someone explicitly named a subdomain after Adminer on a staging environment.

Verification

Every match still needs manual verification. The workflow after getting a hit:

1. Confirm the service is actually running — not a WAF returning 200 for everything
2. Check default credentials — with body verification, not just HTTP status
3. Verify the H1 match is real — does the domain appear in that program's scope?
4. Check /.well-known/security.txt if there's no H1 program
Credential check gotcha Posting Adminer credentials to a Next.js app or CloudFront distribution returns HTTP 200 with HTML, just like a successful login would. You have to verify the response body contains Adminer-specific strings like Select database or logout before calling it confirmed. A bare 200 means nothing.

False Positives I Fixed Along the Way

The WAF problem is real. Several "open Redis" and "open MongoDB" findings turned out to be Incapsula or Akamai sitting in front of the port, returning HTTP responses to Redis wire protocol. The tell is _Incapsula_Resource or edgesuite.net in the response — if either is present, the port isn't actually exposed.

The GitHub Pages subdomain takeover check needed similar refinement. A CNAME pointing to <org>.github.io returning 404 isn't a takeover if the GitHub org exists — GitHub protects the namespace for existing orgs. You have to verify the org itself doesn't exist before flagging it.

Finding Programs with Wide Scope

One more useful trick — finding bug bounty programs with the most wildcard scope entries:

curl -s "https://raw.githubusercontent.com/arkadiyt/bounty-targets-data/main/data/hackerone_data.json" | python3 -c "
import json,sys
data=json.load(sys.stdin)
for p in data:
    wildcards = [t['asset_identifier'] for t in p.get('targets',{}).get('in_scope',[])
                 if t.get('asset_identifier','').startswith('*.')
                 and t.get('asset_type') == 'URL']
    if len(wildcards) >= 5:
        print(p['handle'], '—', len(wildcards), 'wildcards')
        for w in wildcards[:3]: print('   ', w)
        print()
"

Wide wildcard scope means more surface area. Programs like Epic Games, Goldman Sachs, and PayPal all have significant wildcard coverage across acquired properties and subsidiary domains — many of which see less researcher attention than their core assets.

// key takeaways

  1. Cross-referencing before verification saves hours. Knowing a target has a bug bounty program before you start digging is the difference between productive research and wasted effort.
  2. Org name matching is too loose. Hostname fragment matching produces far better signal — if "adminer" appears in a subdomain, someone put it there on purpose.
  3. WAFs intercept more than you think. A TCP connection succeeding doesn't mean the service is exposed. Always verify with a protocol-appropriate probe and check for WAF fingerprints in the response.
  4. HackerOne's public endpoint is gone. Use bounty-targets-data instead — it's better anyway since it includes structured scope data with explicit wildcard domains.
  5. The false positive rate on automated credential checks is high. Body verification against service-specific strings is mandatory before calling anything confirmed.
